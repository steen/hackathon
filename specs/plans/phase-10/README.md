# Phase 10 — End-to-end encryption for channels and DMs

**Parent epic:** #975 — _Phase 10: End-to-end encryption for channels and DMs_
**Decision log:** `lt -p e2e-encryption 3` (numbered §1–§10, locked-in defaults L1–L41, residual risks R1.1, R1.2)
**Contract sub-issue:** #976 — this directory + the PRD addendum

## Overview

Phase 10 lands the encryption epic in three orthogonal surfaces, all gated by a single migration:

1. **User identity keypairs** — every user publishes `(box_pubkey, sign_pubkey)` derived deterministically from a passphrase via Argon2id + HKDF-SHA256 + NaCl seed-keypair operations. Server stores only the public halves.
2. **Explicit channel membership + public/private split** — `channel_members` becomes the wrap-list source of truth. `channels.is_public` distinguishes auto-add (`#general` and any future public channel) from invite-only (every other channel). Every membership-add carries an Ed25519 inviter-signature so a rogue server cannot inject itself into a private channel.
3. **End-to-end-encrypted message envelopes** — `messages` and `dm_messages` drop their `body TEXT` columns and gain a seven-column envelope (`cipher_suite`, `key_generation_id`, `nonce`, `ciphertext`, `sender_sign_pubkey`, `signature`, `client_created_at`). Per-recipient root-key wraps live in `channel_keys` / `dm_conversation_keys`. The server has no plaintext path.

The PRD addendum (§4, §9, §10, §11, §13) and this directory together are the full contract. Every Phase 10 sub-issue cites either a PRD section, a `specs/plans/phase-10/*.md` file, or a numbered decision-log entry (`§N` or `LN`) for any cross-feature shape — sub-issue bodies do NOT copy decision-log text.
