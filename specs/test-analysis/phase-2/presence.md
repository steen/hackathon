## Findings

### Coverage notes

- **Order matters in the handler.** `wsapi/handler.go:186-194` puts `Subscribe(channel, sub)` *before* `AddPresence(userID)` so the joining user's own connection is in the broadcast target set when the join fan-out runs. Without this ordering, the first connection for a user would never see its own join event ‚Äî confusing for clients that use the join frame as a "you're online" handshake. Documented in the inline comment; covered indirectly by `TestPresenceJoinFiresOnFirstConnection` (which does observe the join from the same connection that triggered it via a second observer subscriber).
- **`BroadcastAll` is the right fan-out for presence.** Per-channel broadcast would scope presence to subscribers of one channel; an online indicator is logically global. `hub.go:106` (`BroadcastAll`) walks every channel's subscriber set ‚Äî slightly more work than a dedicated "all clients" set, but no client-state to maintain and no risk of a presence event leaking *outside* the hub. Tradeoff is documented by the `TestBroadcastAllReachesEverySubscriberOnce` test (asserts subscriber-set deduplication so a multi-channel subscriber doesn't see the presence frame twice).
- **Username lookup tolerates row deletion.** `lookupUsernames` (`presence_handlers.go:71-106`) returns one entry per *input ID*, with empty `username` when the row is missing. The doc-comment at the type level (lines 14-17) explains the rationale: the user is still "online" by the hub, so dropping them silently would mismatch the WS-side reality. `TestPresenceTolerateMissingUserRow` pins this contract.
- **gosec G202 suppression is the right call.** The `IN (?,...)` placeholder string is built from a fixed alphabet (`?,`) ‚Äî no user input enters the SQL string; every ID is bound through the parametrized `args` slice. The inline `//nolint:gosec` comment with explanation (`presence_handlers.go:80-84`) keeps a future reader from re-flagging it.
- **Stable response order.** `presence_handlers.go:58-63` sorts by `(username, id)` so the JSON response is deterministic. Important because clients render the list directly; an unsorted response would shuffle on every poll. Test coverage is implicit (`TestPresenceListsHubUsersWithUsernames` builds an ordered expected slice).
- **Refcount overflow is not a real concern.** A `map[string]int` that increments on every WS connection would only overflow at 2^31 connections per user ‚Äî outside the threat model. No defensive cap needed.

### Cross-feature observations

- **`feature-ws-userid-binding-and-channel-existence-check` (gap-D) is the prerequisite that landed first.** Before that PR, `wsapi/handler.go` had `_ = userID` and `connState{userID, channel}` did not exist. Presence relies on the userID being on the connection state ‚Äî gap-D made this feature feasible. The `Prerequisites` section of `50-feature-presence.md` calls this out explicitly. Tracked-and-closed.
- **`ts-api-client-package` AC-6 should re-promote on the next tick.** Phase-2 PR #82 (in flight, not on main yet at this SHA) marked AC-6 (presence event surface) as *partial* because the wire format was the client's *guess* at what this feature would emit. Now that the server-side wire format is real (`{type:"presence", data:{kind:"join"|"leave", user_id:...}}`), the client's `PresenceEvent` type (`packages/api-client/src/types.ts:40-50`, defined as `{type:"presence", data:{kind:"join"|"leave", user:User}}`) **disagrees on the data shape.** The server emits `{kind, user_id}` (just the ID) while the client expects `{kind, user:User}` (the full user record). This is a real schema-drift bug worth flagging ‚Äî the client decoder will surface a `PresenceEvent` with `data.user` undefined. Either the server should embed the full `User` (consistent with the spec phrase "online users" and the REST endpoint's `{id, username}` shape) OR the client should narrow `PresenceEvent.data` to `{kind, user_id}` and let callers join against `GET /api/presence`. The findings doc for ts-api-client (in PR #82) names AC-6 as partial pending this verification ‚Äî when PR #82 merges and the next test-watch tick runs, this AC needs to flip to **partial-with-schema-drift** rather than covered, with the discrepancy spelled out so the maintainer can pick which side moves.
- **`packages/go-client/ws.go::EventTypeMessage` constant** (`= "message"`) doesn't have a sibling `EventTypePresence = "presence"` constant. The Go client's `decodeEvent` (`ws.go:96-113`) only special-cases `EventTypeMessage` for the typed-payload path; a presence frame would land in `Event.Type = "presence"` with `Event.Message` nil and `Event.Raw` populated. That's a forward-compat success ‚Äî clients can branch on `Event.Type == "presence"` and decode `Event.Raw` themselves ‚Äî but a typed `Event.Presence *PresenceEvent` field would be cleaner. Spec follow-up rather than a coverage gap.

## Recommendations

1. No new tests added by this run ‚Äî coverage is comprehensive across the 4 covered ACs at the unit + integration boundary, including the load-bearing concurrent test (`TestPresenceConcurrent`) and the multi-connection refcount semantics (`TestPresenceMultipleConnectionsCountAsOneUser`).
2. **AC-4 stays deferred** until the web app and CLI presence consumers ship (`40-feature-web-app` and a future CLI extension). When either lands, re-evaluate at the new SHA.
3. **Schema-drift flag for cross-feature follow-up:** the TS api-client's `PresenceEvent.data` shape is `{kind, user:User}` while the server emits `{kind, user_id}`. Pick one ‚Äî embedding the full `User` matches the REST endpoint's shape (`{id, username}`) and avoids a join, but doubles the per-event payload size. The current server choice (just the ID) is the lighter-weight option but requires the client to maintain a userID‚Üíusername map. **Production change required either way; out of test-agent scope.** Mention to the maintainer when reviewing PR #82.
4. **Spec follow-up (out of test-agent scope):** clarify in `50-feature-presence.md` whether the presence frame should embed the full user record or just the ID. The current ambiguity is what produced the schema-drift between the two clients.
