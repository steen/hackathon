# /test-analyze — feature → AC → test coverage analysis

You scan the repo's feature specs, extract acceptance criteria (ACs), and check whether each AC has a corresponding test. You write structured findings to markdown.

## Arguments

`$ARGUMENTS` — optional path to operate in (a worktree). Defaults to the current working directory. Treat this path as `$ROOT` for the rest of this skill.

## Inputs

- Feature specs: `$ROOT/specs/plans/phase-*/feature-*.md`. Each has an `## Acceptance criteria` section with bulleted criteria.
- Existing tests:
  - Go: `$ROOT/**/*_test.go`
  - TS / vitest: `$ROOT/**/*.{test,spec}.{ts,tsx,mts,cts,js,jsx,mjs,cjs}` and `$ROOT/**/*_test.{ts,tsx,...}`
- Source code: `$ROOT/apps/**`, `$ROOT/packages/**`. Read to understand what's actually implemented.

Skip `node_modules/`, `.claude/worktrees/` (other worktrees), `dist/`, `bin/`.

## AC ID convention

The existing tests anchor to AC IDs like `AC-1`, `AC-2`, `AC-3` (see `tests/scaffold/go_module_test.go` and `tests/scaffold/pnpm_workspace_test.ts`). Use the same convention.

For each feature spec:
1. Locate the `## Acceptance criteria` section.
2. Number each top-level bullet `AC-1`, `AC-2`, … in order. The spec itself may not write the IDs — that's fine; you assign them positionally and they become the canonical reference for tests.
3. If the bullet is a sub-bullet of another, treat it as part of the parent AC, not a separate AC.

## Coverage detection

For each `AC-N` of a feature:
- Search test files for the literal token `AC-N` AND the feature slug. A match in any test file = covered.
- Also count covered if the test file's `describe`/`TestACN_…`/comment contains the AC text verbatim or a clear paraphrase of the AC's main verb + noun (e.g. AC says "pnpm-workspace.yaml exists at repo root" → test name "pnpm-workspace.yaml exists at repo root" counts).
- Otherwise: missing.

When uncertain whether a test really exercises the AC vs. just mentions the ID in a comment: mark it as **partial** rather than covered, and note the doubt in the findings.

## Implementation status detection

For each feature, infer whether the production code exists yet:
- For Go features (server, CLI): check `$ROOT/apps/<name>/` for files beyond `doc.go`. If only `doc.go` exists, status is `stub`.
- For workspace/scaffold features: check the relevant config files (`go.mod`, `pnpm-workspace.yaml`, `package.json`).
- Status values: `implemented`, `partial`, `stub`, `unknown`.

ACs whose implementation is `stub` get marked **deferred** in findings — the corresponding test should be a skipped test (so it shows up but doesn't fail) until the impl exists.

## Output: per-feature findings doc

For each feature spec at `specs/plans/<phase>/feature-<slug>.md`, write:

`$ROOT/specs/test-analysis/<phase>/<slug>.md`

Use this exact template:

```markdown
---
feature: <slug>
phase: <phase>
analyzed_at: <ISO-8601 timestamp>
analyzed_commit: <full SHA of HEAD>
implementation_status: <implemented|partial|stub|unknown>
total_acs: <int>
covered: <int>
partial: <int>
missing: <int>
deferred: <int>
---

# Test analysis: <feature title>

**Spec:** `specs/plans/<phase>/feature-<slug>.md`
**Implementation status:** <status> — <one-line evidence, e.g. "apps/server contains only doc.go">

## Acceptance criteria

| AC | Statement | Status | Test reference |
|----|-----------|--------|----------------|
| AC-1 | <verbatim AC text> | covered | `tests/scaffold/go_module_test.go:TestAC1_…` |
| AC-2 | <verbatim AC text> | missing | — |
| AC-3 | <verbatim AC text> | deferred | impl is stub |

## Findings

### Missing tests
For each missing AC, one short paragraph:
- What the AC asserts.
- What kind of test would verify it (Go unit, vitest unit, system test).
- Where it should live (file path).
- Any setup it needs (binaries, fixtures, env).

### Deferred tests
For each deferred AC: same structure, plus the impl gap that must close before the test can run live.

### Partial / suspect coverage
For each partial: what the existing test covers vs. what's missing.

## Recommendations

A short bulleted list of concrete next steps for the test-implement skill.
```

Write `$ROOT/specs/test-analysis/README.md`:

## What to return to the caller

When invoked by `/test-watch`, after writing files, emit a single chat line in this exact format so the orchestrator can parse it:

```
test-analyze: features=<F> total_acs=<T> covered=<C> partial=<P> missing=<M> deferred=<D>
```

Plus the list of findings file paths, one per line, prefixed with `findings: `.

## Things you must NOT do

- Do not modify any file outside `$ROOT/specs/test-analysis/`. Code stays untouched here.
- Do not invent ACs that aren't in the spec, and do not renumber existing AC IDs that tests already reference.
- Do not skip features whose spec has no `## Acceptance criteria` section — write a findings doc that says so explicitly (`total_acs: 0`, recommendation: "spec lacks AC section, please add").
- Do not write findings for files outside `specs/plans/phase-*/feature-*.md` (e.g. don't analyze the phase overview docs).
