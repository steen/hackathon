# Hackathon repo — handoff summary

**Goal**: Discord-Lite MVP per `specs/PRD.md`. Phase plans live under `specs/plans/`. Each phase markdown has a `## Deliverables` checkbox list (this is the source of truth for "what's done"); each `feature-<slug>.md` has a `**Status:**` field (`planned` / `done (PR #N)`).

## Repo conventions

- `CLAUDE.md` is the working-agreements file. Read it. Highlights: no fabrication; mark verified vs assumed; no preamble or trailing summaries; default to no comments; single-module Go (no `github.com/...` paths).
- Test names should carry their requirement ID where the plan defines one (`AC-X.Y`, `US-N`, or `FR-N.M`).
- API responses use envelope `{ ok, data, error }`.
- Files < 800 lines.
- Validation at boundaries (user input, API responses, file content).
- Parameterized SQL only.
- Repository pattern for data access.

## Patterns that worked

- One PR per feature, base `main` (not stacked) — stacked PRs broke when intermediates got squash-merged and their branches deleted.
- `gh` + `git` are aliased through `rtk` for compact output (see CLAUDE.md "RTK" section). Optional but recommended.
- Final test gate **after** all implementation work is the authoritative pass/fail signal — don't trust an LLM's "tests pass" claim without actually running the suite.
- If a PR comes back with broken state, prefer pushing to `<branch>-clean` and opening a fresh PR over force-pushing the original.
