## Operational lessons (this session)

### Cross-process worker collision

Running the phase-loop on multiple Claude Code sessions simultaneously — different machines, different conversations — produced two collisions before the lock was added: #235 and #269 each had a worker on this machine push first, while a sibling worker on another machine had already opened a PR. Workers handled it correctly (pushed back as duplicates), but the wasted work was avoidable.

Codified as the `in-progress` GitHub label, set on a sub-issue at claim time and dropped on PR-open. Phase-loop §4 treats it as STRICT cross-process eligibility — never strip a label you didn't set in the same tick. The matching PR is #370. Workflow `.github/workflows/remove-in-review-label.yml` extended to strip on issue close as a backstop.

The `in-review` label served the same role for pr-review-loop and predates this phase; #295's session is when the parallel `in-progress` lock was introduced for the worker side.

### 100 sub-issue cap

Hit GitHub's per-parent sub-issue cap mid-phase. Workers and reviewers continued filing follow-ups, but several (#390, #391, #393, #394, #395, #396) couldn't be attached as native sub-issues; they were filed as standalone issues with a `Refs #295` comment cross-link. Phase-loop's priority order is the native sub-issue listing — orphaned `Refs` issues are not visible to the loop, so they need manual rehoming if they should be picked up by phase-3 work. (#386 is the one that landed cleanly under #63.)

Codified informally — workers/reviewers note the cap in their report and link via comment as a fallback. No skill change yet; the cleaner fix is probably "once #295 closes, free-up moves capacity to #63" but that's a curation decision.

### Stagger subagent dispatches

Multi-agent ticks that fired 3+ Agent calls in a single tool-call block produced timeout clustering: roughly half the agents would die with `Stream idle timeout - partial response received`. Single-dispatch ticks finished cleanly. Hypothesis: model-side concurrency contention on the API stream.

Codified in memory `feedback_stagger_dispatch.md` — dispatch up to the per-tick cap (3 phase-loop, 2 pr-review-loop) only when total in-flight is below ~5; throttle to 1 in-flight when above. The harness blocks `sleep N && cmd` chains, so within-tick stagger via Bash sleep is unavailable; instead use 1 dispatch per tick + 270s wakeup. Tradeoff: lower throughput, but no half-failed batches to clean up.

### Red CI auto-fix

User instruction "whenever you encounter a PR where CI doesn't pass, spawn a one-off agent to fix the issue" — supervisor now spawns a `general-purpose` fix agent on any failing CI rather than surfacing to the user. Pre-existing flakes go in their own drive-by PR (per CLAUDE.md), not on the failing PR's branch. The `#315 → flake-fix → #324` sequence was the template that prompted the rule. Memory `feedback_red_ci_spawn_fix_agent.md`.

### Stale-issue pushback pattern

When a worker discovers the AC is already covered by an existing test (we drafted #295 against a slightly stale view of `tests/e2e/`), it pushes back rather than re-implementing. The supervisor closes the sub-issue with a comment in the established `#206 / #213 / #223` style — quote the AC, point to the existing test file with line ranges, link the worker's report. Roughly a dozen sub-issues closed this way during the phase. The pattern lives in `phase-loop` §11 "Stale-issue pushback".

### Continuous task cleanup

User instruction "clean up the finished items from your todo list continuously" — TaskUpdate to `completed` is followed immediately by TaskUpdate to `deleted`. Memory `feedback_continuous_task_cleanup.md`. (The supervisor's task list was reaching 60+ entries before this.)

### "Always 1 phase-loop worker"

User clarified that phase-loop's job is to keep the worker pipeline non-empty. Per-tick caps are maximums, not targets — but the supervisor must dispatch at least one worker if zero are in flight, even if reviewer / fix agents are running. Encoded in `feedback_stagger_dispatch.md` "Edge case — keep at least 1 phase-loop worker active".

### ANSI in idle banners

Investigated why phase-loop / pr-review-loop idle-banner ASCII art renders without color. Conclusion: Claude Code's terminal renderer strips ESC bytes from agent output. Not feasibly fixable without an Anthropic-side renderer change. Banners still serve as visual idle-tick markers — the prose is the load-bearing part — but the muted-grey palette is aspirational.
